# StockAgent 정밀 분석 (talon 설계 관점)

- 원본: https://github.com/MingyuJ666/Stockagent (논문 arXiv:2407.18957, ACM TIST 게재)
- 분석 대상 커밋: `e2a9c05` (README 갱신), 클론 시점 최신 HEAD
- 한 줄 결론: **트레이딩 시스템도 백테스터도 아니다.** LLM 에이전트가 "투자자처럼" 행동할 때 거시요인이 시장에 미치는 영향을 관찰하는 학술용 다중에이전트 시뮬레이터(ABM)다. talon에 그대로 쓸 것은 없고, 패턴 2~3개만 빌린다.

## 1. 무엇을 하는 코드인가

50명의 LLM 트레이더 에이전트가 가상 거래소에서 서로 매매하며, **주가가 실제 데이터가 아니라 에이전트들의 주문 흐름에서 내생적으로 창발**하도록 만든 시뮬레이션이다. 핵심 셀링포인트는 "테스트셋 누출 방지" — 종목을 A/B/C/D로 익명화하고 재무제표를 합성 문자열로 넣어, LLM이 학습 때 본 실제 티커/가격을 재현하지 못하게 한다. 즉 실제 시장 예측이 아니라 *행동 패턴 연구*가 목적이다.

## 2. 모듈 구조와 실행 흐름

- `main.py` — 시뮬레이션 루프(이벤트 순차형). 일자 → 세션(3회) → 에이전트 순서 랜덤 셔플 → 주문 매칭.
- `agent.py` — `Agent` 클래스. LLM 호출(`run_api`, gpt/gemini 분기), 대출/매매/파산/포럼글/다음날예측 로직.
- `secretary.py` — **LLM이 아니라 순수 파이썬 검증기.** `check_loan/check_action/check_estimate`가 LLM JSON 출력의 스키마·제약(현금 초과 매수, 보유 초과 매도 등)을 검사한다. `run_api`/`get_response` 메서드가 있으나 `openai.api_key=""`로 하드코딩됐고 실행 경로에서 호출되지 않는다.
- `stock.py` — 종목 상태와 가격 갱신(세션 마지막 체결가 = 새 가격).
- `util.py` — 모든 "데이터"(합성 재무제표, 특별이벤트, 대출/상환 상수, 논문 Table의 이상가격 경계)와 설정 상수.
- `record.py` — 매 레코드를 매번 `pd.read_excel → concat → to_excel`로 저장(O(n²) 성능, 연구용).

일일 4단계: **Initial**(포트폴리오 랜덤 초기화, 성향 부여) → **Trading**(대출 결정 → 3세션 매매) → **Post-Trading**(포럼 글 게시, 분기 재무제표 공개, 다음날 예측) → **Special Events**(78일차 지준율 인하, 144일차 금리 인상 메시지 주입).

## 3. 주문 체결 / "백테스팅" 설계

**백테스팅 엔진이 없다.** 실제 과거 시세 재생, 벡터화/이벤트 리플레이, 수수료·슬리피지·룩어헤드 처리가 전무하다. 대신 `handle_action`이 소박한 지정가 매칭을 한다:

- 매수 주문은 반대편 매도 리스트를 순회하며 **가격이 정확히 일치**(`==`)할 때만 체결. 가격-시간 우선순위나 호가 스프레드 개념 없음.
- 매 거래일 시작 시 전일 미체결 주문 전부 삭제. 세션 종료 시 `price = 마지막 체결가`, 체결 없으면 가격 불변.
- 가격은 100% 에이전트 주문에서 창발 → **실제 시장과 상관관계가 0.** 이 시뮬레이터로는 어떤 전략의 수익성도 검증할 수 없다.

리스크 관리는 대출/이자/파산 청산(현금<0이면 강제 매도)과 "대출은 초기자산 초과 불가"뿐. 전략 수준의 손절·포지션 사이징·MDD 통제는 없다. 라이브 트레이딩 어댑터, 브로커 연동, 실데이터 소스 연동은 **전무**하다.

## 4. LLM 오케스트레이션 (talon에 가장 중요한 부분)

- **역할 분담 없음.** 50개 에이전트 모두 동일한 트레이더 프롬프트를 쓰고, 오직 `character`("Conservative/Aggressive/Balanced/Growth-Oriented") 문자열 하나로만 차별화된다. 리서처/애널리스트/리스크 등 전문화된 역할이나 감독자 에이전트가 없다.
- **토론/합의 메커니즘 = 게시판(forum).** 유일한 에이전트 간 소통은 하루 끝에 각자 자유 텍스트를 게시하고, 그 글 묶음(`last_day_forum_message`)이 다음날 모두에게 입력되는 것. 구조화된 토론이나 합의 투표가 아니라 **간접·비동기 정보 전파**다.
- **메모리/반성(reflection)은 매우 약하다.** `chat_history`는 매 거래일 시작 시 초기화(하루치만 유지). `next_day_estimate`로 각자 내일 행동(매수/매도/대출 yes/no)을 자기예측하지만 Excel에 기록될 뿐 다음날 의사결정에 구조적으로 재주입되지 않는다.
- **프롬프트 설계**: `procoder`(PromptCoder)로 `NamedBlock/NamedVariable/Collection`을 조립, `##` 인덱싱. 출력은 JSON 강제. **generate→validate→repair 루프**가 핵심: LLM이 JSON을 뱉으면 secretary가 형식·제약을 검사하고, 실패 시 오류 메시지를 붙여 최대 3회 재요청한다.
- **LLM 호출량 = 비용 폭탄.** 에이전트 1명의 하루 = 대출결정 1 + 매매 3세션 + 다음날예측 1 + 포럼글 1 ≈ **6콜**(+재시도 최대 3배). 50명 × 10일 ≈ **최소 3,000콜, 재시도 포함 1만+**. 논문 설정(210일)이면 **6만 콜 이상.** 전형적인 종량제 토큰 과금 전제 설계다.
- 제공자: `agent.py`는 OpenAI·Gemini 네이티브 SDK만 라우팅. `util.py`에는 yunwu.ai 프록시, Claude/DeepSeek 이름이 있으나 `run_api`에 배선돼 있지 않다(미완성).

## 5. 유지보수 상태 — 심각한 경고

이 레포는 **절반만 리팩터된 깨진 상태**다.
- `util.py`·`prompt/agent_prompt.py`는 4종목(A/B/C/D) + 실험 시드/러너 + 비-LLM 베이스라인으로 갱신됐는데, `main.py`·`agent.py`·`secretary.py`는 여전히 **2종목(A/B) 구버전**이다.
- `DECIDE_BUY_STOCK_PROMPT`는 `{stock_c_price}` 등 8개 C/D 변수를 참조하지만 `agent.plan_stock`의 inputs엔 A/B만 있어 `format_prompt`가 KeyError로 깨진다. **HEAD는 그대로 실행되지 않는다.**
- `plan_loan` 재시도가 `check_loan(date, resp)`로 잘못된 시그니처 호출(원래 `resp, max_loan`). 대출타입 프롬프트("1/2/3년")와 `LOAN_TYPE_DATE=[22,44,66]일`·`LOAN_TYPE=one/two/three-month`가 서로 모순.
- `util.py`의 미래형 모델명(gpt-5.5, Claude-4.6-Sonnet 등)은 실 배선과 무관한 플레이스홀더. 테스트 코드 없음. 커밋 이력 사실상 없음.

연구 재현용 스냅샷일 뿐, 소프트웨어로서의 신뢰도는 낮다. **이 코드베이스를 포크하지 마라.**

## 6. talon 적용성 평가

### 그대로 채택할 것 — **없음**
트레이딩·백테스팅·라이브·리스크 어느 축도 talon에 재사용 가능한 완성 컴포넌트가 아니다.

### 패턴만 빌릴 것 (핵심 수확)
1. **Secretary 검증기 패턴(강력 추천).** LLM 자유생성과 결정론적 스키마·계좌상태 검증을 분리하고, 실패 시 오류를 되먹여 유한 재시도로 자가교정. talon은 이걸 그대로 차용해 "LLM이 제시한 주문 JSON"을 토스 잔고·보유수량·호가단위·±30% 상한가/하한가·최소주문금액에 대해 검증한 뒤에만 사용자에게 제시해야 한다. **LLM 출력을 절대 그대로 신뢰하지 않는다**는 원칙.
2. **구조화된 결정 스키마.** `{action_type, stock, amount, price}` 같은 좁은 JSON 계약 + 형식검사는 talon의 LLM→주문근거 인터페이스 설계에 직접 유용.
3. **일일 단계 루프.** 장전 계획 → 장중 세션 → 장후 반성/게시의 위상 구조는 단타/스윙 일과에 잘 매핑된다.
4. **경량 메모리/저널 = 1인용 forum.** 게시판을 "단일 에이전트의 일일 리서치·테제 저널"로 축소해, 어제의 판단과 결과를 다음날 프롬프트에 재주입하는 반성 루프로 재해석할 수 있다.
5. **성향 조건화**는 talon 1인용에선 불필요하나, 사용자 리스크 프로파일로 프롬프트를 조건화하는 아이디어로만 참고.

### 참고만 할 것
- 익명화(A/B/C/D)로 LLM의 사전지식 누출을 막는 방법론 — talon 백테스트 설계 시 "LLM이 종목·날짜를 인지해 미래를 커닝하는 것"을 어떻게 차단할지에 대한 개념적 힌트. 실코드는 무관.

### 피할 것과 이유
- **창발형 ABM을 백테스트로 오해하지 말 것.** 실제 시세와 무관하므로 전략 검증 불가. talon 백테스트는 별도로, 실 KRX/미국 시세 기반 이벤트드리븐 엔진(체결모델·수수료·슬리피지·룩어헤드 차단 포함)으로 처음부터 만들어야 한다.
- **종량제 다중콜 설계**(수천~수만 콜). talon의 구독제 LLM(Claude Max / Codex Pro) 제약과 정면충돌.
- 정확일치 지정가 매칭, 수수료/슬리피지 부재, 반토막 리팩터 코드 품질.

### 한국 시장 적용성
Korea 특화 요소가 **전혀 없다.** 원화, T+2 결제, ±30% 가격제한폭, 호가단위, 장 시간, 세금·수수료 어느 것도 없다. 전부 talon이 토스 OpenAPI 위에서 새로 구축해야 한다. 빌려올 것은 오케스트레이션 패턴뿐.

### 구독제 LLM 제약과의 궁합 — **나쁨**
StockAgent 설계 철학(50 에이전트 × 수십 세션 × 매턴 LLM 콜)은 값싼 토큰 과금을 전제한다. talon은 정반대여야 한다: **에이전트 1개**, LLM은 리서치·뉴스 종합·매매근거 서술 같은 저빈도 고가치 추론에만 쓰고, 검증·매칭·지표계산·백테스트는 전부 결정론 코드로 오프로드. StockAgent에서 talon이 얻는 단 하나의 이식 가능한 통찰이 바로 이 "LLM은 추론, 코드는 검증" 분업(secretary 패턴)이다.
