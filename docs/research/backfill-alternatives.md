## 1. 배경 — KRX 차단

talon은 한국 주식(KOSPI/KOSDAQ) 단타·스윙용 개인 퀀트 시스템으로, 백테스트를 위해 최근 10년치 전종목 일봉(OHLCV·거래대금·시가총액·상장주식수, 가능하면 상장주식수·수급)이 필요하다. 데이터 정합성과 룩어헤드 안전성이 최우선이며 상폐 종목 포함(생존편향 방지)이 중요하다.

**[사실]** 2025-12-27부로 KRX 정보데이터시스템이 회원제 'KRX Data Marketplace'로 전면 개편되며 로그인·인증키를 필수화했다. 그 결과 비인증 스크래핑 경로(`data.krx.co.kr/getJsonData.cmd`)가 사실상 폐지되어 pykrx의 전종목 조회가 400 'LOGOUT'으로 실패한다. 프로젝트 메모의 '2026-07-11 차단'은 팀 인지 시점이며, 근본 원인의 실제 전환일은 2025-12-27이다. 이는 일시적 IP 차단 사이클이 아니라 구조적·영구적 정책 변경이므로 원복 전망은 낮다.

**[사실 · 실측]** `uv run talon doctor --live` 및 pykrx 직접 호출로 재현: 전종목 스냅샷 `get_market_ohlcv(day, market="ALL")`·`get_market_cap`는 직전 거래일(2026-07-10)에 대해 4/4 실패(`KeyError: None of ['시가','고가','저가','종가'] are in the columns`). 반면 **개별종목 조회는 정상**(삼성전자 10년 2453행 반환).

> **[2026-07-13 정정 — 위 진단은 절반만 맞았다]**
> 차단 자체는 사실이나, **"pykrx는 죽었다"는 결론은 틀렸다.** 진짜 원인은 **talon이 pykrx 1.0.51(2025-05 릴리스)에 고정돼 있었다는 것**이다. KRX 회원제 전환은 2025-12-27 — 즉 **우리 pykrx는 차단 사태보다 7개월 앞선 버전이라 애초에 회원제 로그인을 할 줄 모르는 코드**였다.
>
> **pykrx 1.2.8(2026-05-04)에 KRX 로그인이 정식 탑재**되어 있다(`pykrx/website/comm/auth.py`, `data.krx.co.kr` 세션 로그인, 환경변수 `KRX_ID`/`KRX_PW` 자동 인식, 세션 1시간 만료·자동 갱신). 버전을 올리고 KRX Data Marketplace 계정(= Open API 키를 받은 그 계정)을 넣자 **당일(T+0) 전종목 스냅샷이 즉시 복구됐다** — 2026-07-13 기준 2,873종목, 시가총액·상장주식수 포함.
>
> **교훈:** 소스가 죽었다고 판정하기 전에 **클라이언트 버전이 사건 이후 것인지부터 확인할 것.** 우리는 7개월 묵은 클라이언트로 새 서버를 두드리며 "서버가 죽었다"고 6개월간 기록해뒀다.

## 3. 후보별 상세 (실측 결과 포함)

### 3.1 FinanceData/marcap — 백필 본체 (1순위) ⭐

**[사실 · 실측 PASS]** GitHub 공개 리포의 연도별 parquet(`raw.githubusercontent.com/FinanceData/marcap/master/data/marcap-YYYY.parquet`, 1995~2026). 계정·키 불필요. 오늘 재확인: 2026 파일 HTTP 200·13.7MB, 최신 커밋 `auto-update dataset` 2026-07-10.

4대 기준 전부 통과:
- **(a) 필드 완비:** 18컬럼 `[Code,Name,Close,Dept,ChangeCode,Changes,ChangesRatio,Volume,Amount,Open,High,Low,Marcap,Stocks,Market,MarketId,Rank,Date]`. 요구 필드 Open/High/Low/Close/Volume + **Amount(거래대금) + Marcap(시가총액) + Stocks(상장주식수)** 전부 존재. 날짜별 스냅샷 1파일로 talon의 `DAILY_SNAPSHOT_SCHEMA`와 `MARKET_CAP_SCHEMA`를 동시 충족.
- **(b) 상폐 보존(생존편향 없음):** 한진해운(117930)이 marcap-2016에 246거래일 보존, 2026 파일엔 부재. 상장 당시 날짜에 그대로 남음.
- **(c) 최신성:** marcap-2026 max Date 2026-07-09(오늘 T-2, 직전 거래일까지). 하루당 종목수 중앙값 2,880.
- **(d) 원주가 확정:** 삼성전자 2018-05-04 50:1 액면분할 경계에서 Close 265만→5.19만(51배 급락), Stocks 정확히 50.0배 급증 → **원주가(비수정) 저장 확정.**

**[사실 · 크로스체크 실측]** marcap × 네이버 siseJson 정합성: (a) 삼성 분할계수 50→1과 경계일 2018-05-04가 두 소스 정확 일치, (b) **Marcap = Close × Stocks가 2018 전체 파일(567,521행) 100% 정확** → 과거 시총 완전 재현, (c) Amount는 밴드 내이며 2018-05-31 저가 이탈(내재 VWAP < 정규장 저가)이 오히려 실제 KRX 체결 거래대금(합성 아님) 근거.

**[사실 · 소멸코드 조인 실측]** 2016 소멸코드 356건이 FDR KRX-DELISTING로 100% 설명(unexplained 0). 순수상폐 149/149에서 marcap 마지막 거래일 == 정리매매 종료일(ArrantEndDate) 정확 일치.

**리스크:** 원주가라 수정주가 재구성 필수 / 명시적 LICENSE 없음(개인 비상업 저위험) / 향후 일일 갱신은 유지보수자 CI(KRX 의존, 과거분은 불변) / 수급·펀더멘털 없음.

출처: https://github.com/FinanceData/marcap · https://financedata.github.io/marcap/

### 3.2 네이버 금융 siseJson — 수정주가 짝 (2순위)

**[사실 · 실측 PASS]** `api.finance.naver.com/siseJson.naver`, 무인증·무키. 오늘 재확인 HTTP 200/47ms.
- 깊이: 삼성 5310행(2005~현재), 인위적 캡 없음.
- **수정주가:** 2018 분할 경계 연속(분할 전 원주가 ~265만이 5.19만으로 1/50 조정) → 백테스트 적합.
- 상폐 보존: 한진해운 1778행(→20170306), 웅진에너지 2445행(→20200529). 존속 티커(쌍용차→KG모빌리티)는 현재까지.
- 필드: `[날짜,시가,고가,저가,종가,거래량,외국인소진율]` — **거래대금·시총·상장주식수 없음**(marcap로 보완).
- 거래정지 아티팩트: O=H=L=0·Volume=0·종가 캐리(상폐 종목·삼성 분할 재편기 모두) → 전처리 필수.

marcap이 원주가라 발생하는 수정주가 공백을 이 소스가 메운다. talon이 이미 FDR DataReader 백엔드(`fetch_symbol_history`)로 이 경로를 사용 중.

**리스크:** 시총/거래대금/상장주식수 부재 / 비공식 API(스펙 변경·약관 회색지대) / fchart 계열 ~3000봉 캡 / 오래전 비인기 상폐 미보관 가능.

출처: `api.finance.naver.com/siseJson.naver?symbol=005930&requestType=1&startTime=20160101&endTime=20260711&timeframe=day`

### 3.3 FDR KRX-DELISTING (fdr_krx_data_cache) — 상폐 메타데이터 (3순위)

**[사실 · 실측 PASS]** `fdr.StockListing('KRX-DELISTING')` 또는 raw CSV(`.../fdr_krx_data_cache/.../data/listing/delisting/YYYY-MM-DD.csv`). raw 경로는 **KRX 무호출**이라 차단과 무관. 오늘 최신 2026-07-11.csv HTTP 200.
- 4159행, 1960-11-21~2026-07-09, 15컬럼: `Symbol,Name,Market,SecuGroup,Kind,ListingDate,DelistingDate,Reason,ArrantEnforceDate,ArrantEndDate,Industry,ParValue,ListingShares,ToSymbol,ToName`.
- 채움 정도: ArrantEnforce/EndDate(정리매매 개시/종료) 각 ~1,222/1,229, ToSymbol 997, Reason 2758.
- **[사실]** marcap 소멸 356건 100% 설명, 순수상폐 149/149 마지막 거래일 == ArrantEndDate.

**핵심 경고 [사실]:** 소멸 = 상폐가 아니다. 비-부실 기업행위 133건(피흡수합병 44·지주사 완전자회사화 32·자진상폐 32·우선주 소멸 24 등) 중 ToSymbol로 잡히는 건 18건뿐. 우리은행(000030)·메리츠화재(000060)·DL건설(001880)은 ToSymbol 비어있는 완전자회사화라 ToSymbol-only 규칙이면 -100% 오처리됨. → **반드시 Reason 버킷 + ArrantEndDate 유무로 규칙 수립.**

같은 EOD 스냅샷(`data/listing/krx`)은 2026-03-08~ 약 4개월치뿐이라 10년 가격 백필엔 부적합(EOD 일일 폴백 전용).

출처: https://github.com/FinanceData/fdr_krx_data_cache · https://github.com/FinanceData/FinanceDataReader/issues/24

### 3.4 KRX Open API (openapi.krx.co.kr) — 확정본 정본 (2026-07-13 키 발급 후 전면 실측)

게이트웨이 `data-dbg.krx.co.kr`, 헤더 `AUTH_KEY`, 파라미터 `basDd`. 승인받은 엔드포인트: `sto/stk_bydd_trd`(유가), `sto/ksq_bydd_trd`(코스닥), `sto/stk_isu_base_info`·`sto/ksq_isu_base_info`(종목기본정보).

**[사실 · 실측] T+1이다 — 당일 스냅샷 소스로 쓸 수 없다.** 2026-07-13(월) **KST 23:00**, 즉 장 마감 7시간 반 뒤에도 당일 데이터가 없었다(빈 응답). 직전 거래일(07-10)·그 전날(07-09)은 정상. → **EOD(16:40)에서 FDR/pykrx를 대체 못 한다. 이 API의 자리는 "익일 확정본"이다.**

**[사실 · 실측] marcap과 전 필드 완전일치.** 2026-07-09, 교집합 2,652종목에서 시가·고가·저가·종가·거래량·거래대금·시가총액·상장주식수·등락률 **전부 최대오차 0.00**. → 그동안 검증할 방법이 없던 **marcap 10년 백필이 KRX 공식과 소수점까지 같음이 확증**됐다.

**[사실 · 실측] 종목 수 차이의 정체.** KRX 2,765 vs marcap 2,729(교집합 2,652).
- **KRX에만 있는 113종목 → 전부 거래량 0(거래정지).** marcap이 거래량 0 행을 버리는 것뿐. 2017년(42건)~2026년(113건) 10년 내내 같은 패턴. 유니버스가 `거래량>0`을 요구하므로 무해.
- marcap에만 있는 77종목 → 거래대금 최대 7천만원짜리 잡상품. 유니버스 하한(10억) 미달로 무해.
- **[구현 주의]** 따라서 KRX Open API 응답에는 반드시 `(종가>0 AND 고가>0)` 필터를 걸어 marcap/pykrx와 기준을 맞춰야 한다. 안 그러면 reconcile이 매일 113종목을 '신규'로 오인해 무한 교정한다.

**[사실 · 실측 — 기존 '추정' 확증] 생존편향 없음.** 2024-07-15 스냅샷 2,697종목 중 153개가 2026년엔 소멸했고, 그 중 **진성 상폐(terminal) 104건이 당시 종가와 함께 정상 수록**(스마트솔루션즈 11,600원 / 녹원씨엔아이 11,540원 / 한국테크놀로지 334원 …). 나머지는 corporate_action 41 · unknown 8. → **KRX Open API 단독으로 2010년부터 생존편향 없는 백필이 가능하다.**

**[사실 · 실측] 휴장일 = 빈 응답 → 거래일 사전으로 쓸 수 있다.** 2026-06-03(제9회 지방선거) 0건 / 06-04(정상) 948건 / 06-07(일요일) 0건. → `markets/kr.py`의 수동 딕셔너리 `CLOSURES_MISSING_FROM_XKRX`를 데이터로 대체 가능(미구현).

**[사실 · 실측] 호출 한도는 헤더에 안 나온다.** 응답 헤더에 rate/limit/quota 관련 필드가 전무 → **남은 쿼터를 프로그램으로 감시할 방법이 없다.** 일 10,000콜은 여전히 커뮤니티 추정치.

**[사실 · 실측] 401은 두 종류이고, 구분이 진단의 핵심이다.**
| 보낸 것 | 응답 |
|---|---|
| 헤더 없음 / 틀린 키 / 틀린 헤더명 | `{"respMsg":"Unauthorized Key"}` |
| 유효한 키, 미승인 서비스 | `{"respMsg":"Unauthorized API Call"}` |

메시지가 다르므로 **"키가 틀렸나 / 서비스 신청을 안 했나"를 응답만으로 판별할 수 있다.** 키 발급과 **서비스별 이용 신청은 별개**이며 승인에 1~2일 걸린다. 없는 경로는 404에 다른 메시지가 온다.

**[사실 · 실측] 응답 필드 (일별매매정보):** `BAS_DD, ISU_CD(6자리), ISU_NM, MKT_NM, SECT_TP_NM, TDD_OPNPRC, TDD_HGPRC, TDD_LWPRC, TDD_CLSPRC, CMPPREVDD_PRC, FLUC_RT, ACC_TRDVOL, ACC_TRDVAL, MKTCAP, LIST_SHRS` — talon이 필요한 전부. 숫자는 쉼표 포함 문자열.

**[사실 · 실측] 종목기본정보가 유니버스 필터를 정식화해준다 (2026-07-14 구현 완료).**
`SECUGRP_NM`(주권 / 부동산투자회사 / 외국주권 / 주식예탁증권 / 사회간접자본투융자회사 / 투자회사 / 선박투자회사), `KIND_STKCERT_TP_NM`(보통주 / 구형우선주 / 신형우선주 / 종류주권), `SECT_TP_NM`(코스닥 소속부 + 관리종목 · SPAC · 투자주의환기종목).
- 예전 백테스트 유니버스 규칙 `symbol.ends_with("0")`을 KRX 공식 분류와 대조: **잘못 포함 47건**(리츠 23 + 외국주권 12 + 주식예탁증권 9 + 기타 3), 빠뜨림 0건.

**[사실 · 실측 · 2026-07-14 정정] `basDd`가 과거 날짜로 동작한다 — 시점별(point-in-time) 스냅샷이다.**
2016-07-01 885종목 / 2020-06-15 912 / 2024-07-15 955 / 2026-07-10 945로 날짜마다 다르고, 상장주식수도 그 시점 값이다(AJ네트웍스 2016년 9,364,459주 → 2020년 46,822,295주). 관리종목·SPAC 개수도 시점별로 다르다(2020년 관리 48·SPAC 52 vs 2026년 74·71). → **오늘 분류를 과거에 소급 적용하는 룩어헤드 없이 과거 유니버스를 정확히 재구성할 수 있다.** 이것이 `stock_info`를 일자별 파티션으로 적재하는 근거다.

**[사실 · 실측 · 2026-07-14 정정] 🔴 `SECT_TP_NM`은 코스닥에만 채워진다. 위 "관리종목 74·SPAC 71"은 코스닥 수치다.**
유가증권(`sto/stk_isu_base_info`)은 **945종목 전부 `SECT_TP_NM=""`** 이다(일별매매정보 `stk_bydd_trd`도 동일). 소속부(중견기업부·우량기업부·벤처기업부·기술성장기업부)가 원래 코스닥 전용 개념이고, KRX가 관리종목·SPAC·투자주의환기 표시를 그 필드에 얹어놨기 때문이다. → **KOSPI 관리종목은 KRX Open API로 현재조차 받을 수 없다.**

실측한 사각지대 크기(2026-07-10): FDR 관리종목 108건 중 종목기본정보에 있는 것 44건 = 코스닥 42 + **KOSPI 2**. KOSPI 2건 중 1건(제이알글로벌리츠)은 리츠라 `SECUGRP_NM`에서 이미 걸리므로 **진짜 사각지대는 1건(다이나믹디자인)이고, 거래대금 상위 300에도 못 든다.** 코스닥은 FDR 42건 중 39건을 KRX가 직접 잡는다. KOSPI 스팩은 0건(73개 전부 코스닥)이라 SPAC 누락은 없다.
→ **대응:** 라이브는 FDR `StockListing("KRX-ADMINISTRATIVE")`를 KOSPI 관리종목용 보조 게이트로 유지하되, 예전처럼 실패 시 조용히 `None`을 반환하지 않고 **텔레그램 경보를 띄운다**(`admin-list-unavailable`). 백테스트는 이 목록의 과거 시계열이 없으므로 KOSPI 관리종목을 거르지 못한다 — **알려진 잔여 갭이며, 라이브가 백테스트보다 엄격한(=안전한) 방향이다.**

**[해결 · 2026-07-14] 라이브와 백테스트가 이제 같은 규칙을 쓴다.**

분류 규칙은 `quant/universe.py:tradable_stock()` **단 하나**이고, 라이브(`ingest/universe.py`)와 백테스트(`backtest/data.py` → 패널의 `tradable_stock` 컬럼)가 이 함수를 함께 호출한다.

```
SECUGRP_NM == "주권" AND KIND_STKCERT_TP_NM == "보통주"
  AND SECT_TP_NM에 관리종목·투자주의환기종목·SPAC 없음
```

| | 이전 | 현재 (양쪽 공통) |
|---|---|---|
| 보통주 판정 | 라이브=토스 / 백테=`ends_with("0")` | **KRX 공식 `stock_info` 일자별 분류** |
| 관리종목 제외 | 라이브=FDR(무경보 실패) / 백테=없음 | 코스닥은 KRX 공식(시점별) · KOSPI는 라이브만 FDR + **경보** |
| SPAC·투자주의환기 | 없음 | KRX 공식(시점별) |
| 분류 미상 종목 | 통과 | **제외(fail-closed)** |
| 데이터 없을 때 | 조용히 통과 | **하드 실패** (`load_panel` ValueError · 라이브 `SourceError` + 워치독 `stock-info-stale`) |

**유니버스 실제 변화폭(실측):** 거래대금 상위 300 기준 **하루 2~5종목 교체(약 1%)**. 리츠는 거래대금이 작아 애초에 상위 300에 못 들고, 실제로 빠지는 건 맥쿼리인프라(사회간접자본투융자회사), 뉴프라이드·로스웰·윙입푸드·헝셩그룹(외국주권 — 중국계 우회상장), 코오롱티슈진·SNK(주식예탁증권), 그리고 코스닥 관리종목·SPAC·투자주의환기다. **종목 수는 적지만 질이 나쁜 것들이 걸러진다.**

2016-11~2017-04(116거래일) 실측: 옛 규칙만 통과시키던 종목 79개 = 선박투자회사 8(코리아01~04호·하이골드) + 외국주권 12(중국원양자원·완리·헝셩그룹 등 상당수가 이후 분식·상폐) + 리츠 2 + 맥쿼리인프라 + 주식예탁증권 2 + 투자회사 2 + 관리종목·투자주의환기 ~25 + SPAC 14.

**[사실 · 실측] 시점별 분류가 실제로 룩어헤드를 막는다.** 아이앤씨(052860)는 2016-07~2017-03-17 관리종목이었다가 해제됐고, 비에이치아이(083650)는 2017-03-22 투자주의환기에서 해제됐다. **오늘 분류를 과거에 소급했다면 이 종목들의 관리종목 기간을 통째로 매매 가능으로 착각했을 것이다.** 일자별 적재라 지정 기간에만 정확히 빠진다.

**[사실 · 실측] 분류 없는 종목은 fail-closed로 제외되며, 그 정체는 KONEX다.** 일봉(marcap)에는 KONEX가 들어있지만 KRX 종목기본정보는 유가·코스닥만 준다(`knx_*` 엔드포인트는 미승인). 하루 약 100종목이 분류 없이 남지만 **거래대금 10억 이상 (종목,날짜) 281,399쌍 중 제외된 것은 63쌍(0.022%) · 18종목**뿐이다. 076340은 현재도 KONEX 상장이고, 이엘피(063760)는 2017-04-06 코스닥 이전과 동시에 분류가 생겼다. **KONEX는 기본예탁금·유동성 제약이 있는 스타트업 시장이라 단타·스윙 유니버스에서 빠지는 게 맞다 — 옛 `ends_with("0")` 규칙은 이걸 통과시키고 있었다.**

**[운영] 신선도는 T+1이다.** `stock_info`는 reconcile 잡(익일 09:00/13:00)이 채운다. EOD(16:40) 시점에 가진 최신 분류는 전 거래일 것이고, 라이브 유니버스는 `latest_stock_info`로 그걸 쓴다(`universe_info_max_stale_days=10` 초과 시 하드 실패). 당일 신규상장 종목은 다음 날부터 유니버스에 들어온다 — IPO 당일은 어차피 매매 대상이 아니므로 보수적으로 안전한 방향.

**[사실 · 적대검증 REFUTED — 약관 주의]** '상업적 재배포만 제한'이라는 통념은 원문과 배치된다. 공식 약관(OPPINFO002.jsp) 제6조② 비상업 목적만 허용, **제11조② 모든 제3자 제공 금지, 제11조③ 계약 종료 후 데이터 이용 금지**, 제5조 인증키 1년 갱신. 개인 백테스트 로컬 적재는 계약 유효 중 명시적 금지는 아니나(부작위) 재배포·종료 후 이용·영구 보관은 약관 위배 소지.
→ **그래서 marcap을 보관용 정본으로 유지한다.** KRX Open API를 유일 정본으로 갈아타면 키 만료 시 히스토리 전체가 약관상 애매해진다.

**리스크:** 🔴 **서비스별 유효기간이 따로 돈다** — 코스닥 일별매매정보를 실수로 1개월로 신청해 2026-08-13경 만료 예정. 만료되면 코스닥 1,820종목이 401과 함께 조용히 사라진다(`talon doctor --live`가 엔드포인트별로 찍어주므로 거기서 잡힌다). / 2025-12 신설이라 약관·한도 변경 여지.

## 5. 권고 전략 — 조합

단일 소스가 아니라 역할 분담이 최적이다.

```
┌ 백필 본체 (2016~2026, 전종목·전필드·상폐 포함) ─────────────────
│   marcap 연도별 parquet → DAILY_SNAPSHOT + MARKET_CAP 동시 충족
│   Amount→value, Marcap→cap, Stocks→shares (거의 1:1 매핑)
├ 수정주가 재구성 ────────────────────────────────────────────
│   marcap 원주가 × 네이버 siseJson 수정주가 → 종목별 조정계수(이산 계단)
├ 상폐 유니버스·종결 처리 ────────────────────────────────────
│   FDR KRX-DELISTING(raw CSV) — Reason 버킷 + ArrantEndDate 규칙
│   (KIND는 독립 교차검증)
├ 일일 EOD (최근, T 당일) ────────────────────────────────────
│   기존 fdr_krx_data_cache 스냅샷 폴백 유지(marcap은 T-2라 당일 부적합)
│   + 토스 캔들(현재 유지)
├ 정합성 크로스체크 ──────────────────────────────────────────
│   crosscheck.py: marcap(원주가+factor) vs 네이버(수정주가) close/volume
└ 합법 원천 폴백 (등록 권장) ─────────────────────────────────
    KRX Open API — marcap CI 중단 대비 정당 원천, 상폐 포함 실측 검증
```

### 기존 가동 소스와의 정합성
- **FDR 스냅샷 폴백:** 그대로 EOD 일일 폴백으로 유지. `data/listing/krx` 스냅샷(2026-03~)은 최근분만이라 백필용이 아니라 당일 EOD 보강용.
- **토스 캔들:** 분봉 누적·당일 시세는 유지. 토스는 현재 상장 종목만이라 상폐·10년 백필 부적합 — 그 역할을 marcap이 담당.
- **크로스체크 방법:** talon의 `crosscheck.py`가 이미 close/volume 상대차 비교를 구현. marcap 백필 값 vs 네이버 종목별을 factor 반영해 대조하고, 시총은 `Marcap == Close × Stocks` 항등식(2018 전체 파일 100% 검증)으로 자체 검산.

### talon 코드 매핑 (통합 난이도 근거)
- 현재 `ingest/history.py:backfill_daily`는 차단된 `sources/krx_daily.fetch_daily_ohlcv/fetch_market_cap`(pykrx ALL)에만 의존. `sources/marcap_daily.py` 신규 어댑터로 교체하면 날짜 루프·멱등 `write_date`·`BackfillSummary` 구조는 그대로 재사용 → 낮은 난이도.

## 부록 D · 무거래일 행 결손 복원 실행 결과 (2026-07-22)

부록 C의 브리지 실행이 남긴 후보 과제(초저유동 원주가 결손 행 백필)를 실행했다. 커밋 1bab907~8c96dcd.

- **원인 규명 — 결손은 원천 부재가 아니라 자체 유발이었다.** KRX 계열 원천은 무거래일(거래정지·매매 없음)에도 행을 주되 시가/고가/저가를 0으로 기록하고, 종가에는 기준가 또는 기세(무거래 호가 확정가)를 담는다. 4개 일봉 소스 전부(marcap·pykrx·FDR 전종목·KRX Open API)의 `(close>0 & high>0)` 필터가 이 행들을 통째로 버렸다. reconcile의 anti-join 신규 편입도 필터된 official만 보므로 복구 기회가 없었다. 실측 규모: marcap 전 구간(2016-01~2026-07) **272,796행·1,682종목**, 그중 **14,729행은 기세로 종가가 움직여**(예: 076340 2016-09-30 +14.81% 무거래) 다음 거래일 수익률 기준선을 왜곡 — 부록 C의 유형 ③ 유령 사건이 이것이다.
