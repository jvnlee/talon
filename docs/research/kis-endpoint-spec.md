# talon KIS Open API 수집기 최종 스펙 (v1, 2026-07-15 실측 기준)

## 0. 표기 규약 · 공통 전제

**근거 표기**
- **【실측】** = 2026-07-15 장중(약 11:43~11:45 KST) 라이브 호출로 값·구조를 직접 확인한 것
- **【문서】** = KIS 공식 문서·docstring 등 문서로만 확인, 이번 세션 미검증(신호로 쓰기 전 재확인 필요)

**용어 풀이(외래어)**
- PIT = Point-In-Time, "그 시점에 실제로 알 수 있었던 값인가"
- 룩어헤드(look-ahead) = 미래 정보를 미리 훔쳐보는 오염
- 잠정 가집계 = 장중 임시 추정치(확정 아님), 저녁에 확정치로 덮임
- forward-only = 지금부터 매일 스냅샷을 쌓아야만 원천이 생김(과거 소급 불가)
- RPS = 초당 호출 수, 틱(tick) = 최소 시세 갱신 단위
- tr_cont = 연속조회(페이지네이션) 헤더, 잔량 = 호가창 각 단계에 걸린 미체결 주문 수량

**공통 전제**
- 도메인: **실전**. tr_id는 실전=문서값 그대로 사용(모의는 접두 V/T 치환 【문서】). 전 요청 `custtype=P` 고정 【실측】
- 인증 파일(파이썬 로드): 앱키/시크릿 = `~/.talon/env`, 토큰 = `~/.talon/cache/kis_token.json`
- 공통 요청 헤더: `content-type: application/json; charset=utf-8`, `authorization: Bearer <token>`, `appkey`, `appsecret`, `tr_id`(TR별), `custtype=P`, `tr_cont`(단건은 공백). 시세조회 GET은 `hashkey` 불요(미전송에도 rt_cd 0) 【실측】
- 공통 응답: `rt_cd`('0'=정상), `msg_cd`(정상='MCA00000'), `msg1`('정상처리 되었습니다.'), 응답헤더에 `tr_id` 에코 + `gt_uid`(장애 문의 추적용) 【실측】
- 쿼리 파라미터 이름은 **대문자**, output 필드명은 **소문자** 【실측】

## 2. 수집기 설계 지침

### 2.1 토큰 캐시 규칙
- **개인 토큰 유효기간 24h**(`expires_in`=86400, 만료문자열 `access_token_token_expired`) 【실측: 발급 2026-07-15 11:39:59 → 만료 2026-07-16 11:39:59】
- **반드시 파일 캐시 재사용**(`~/.talon/cache/kis_token.json`). 만료 또는 임박 시에만 `/oauth2/tokenP`(POST, body: `grant_type=client_credentials`+appkey+appsecret)로 재발급
- **매 호출·매 잡 재발급 금지** — 유량 소모, `EGW00133`, 기존 토큰 무효화 위험
- **토큰 발급 분당 1회 제한** 【문서】 → 만료 임박에 몰아서 발급하면 실패 리스크. **장전 1회 발급→캐시**, 15:10/15:20 수집은 캐시 토큰만 사용(발급 경로와 수집 경로 분리)
- 앱키·시크릿·토큰 값은 로그/출력에 절대 기록하지 않음(실측 규율 유지)

### 2.2 유량 제한에 안전한 초당 호출 수
- 유량은 **계정(앱키) 단위**: 실전 **20건/초**, 모의 2건/초 【문서】
- 슬라이딩윈도우 경계효과로 20/s 미만에서도 `EGW00201` 가능 → **실효 15/s 이하 권장**
- **권장: 전역(프로세스 간) 스로틀 8 RPS 유지**(기존 config `kis_rps=8.0`은 이미 보수적이라 유지). 병렬 잡·형제 프로세스가 같은 계정을 공유하므로 프로세스 로컬이 아니라 전역 스로틀이어야 함
- 콜 예산: **호가(1.1)·추정수급(1.2)는 종목당 1콜** → 유니버스 순회 시 콜 수 = 종목 수(스로틀 필수). **랭킹(1.3 top30 / 1.4 top100)은 시장당 1콜**로 콜 수 적음

### 2.3 재시도 vs 즉시 중단 에러
| 상황 | 조치 | 근거 |
|---|---|---|
| `EGW00201`(유량 초과) | 지수 백오프 후 재시도 | 【문서】 |
| HTTP 5xx / 네트워크 타임아웃 | 백오프 후 재시도 | 관행 |
| HTTP 401/403 · `EGW00133`(토큰 만료·무효) | **토큰 재발급 후 1회 재시도**(분당 1회 제한 준수). 재발급 후에도 실패면 중단 | 【문서】 |
| `rt_cd != 0` (파라미터 오류·권한 등 그 외) | **즉시 중단**(같은 요청은 재시도해도 동일 실패) | 관행 |
| `rt_cd == 0` 이지만 output 0건(예: NX에서 빈 응답) | 재시도 아님 — 정상 응답이므로 시장구분 코드 선택 오류로 판단, 로깅 후 진행 | 【실측】(NX 0건) |

### 2.4 tr_cont 페이지네이션 사용처
- **본 4개 수집 엔드포인트는 전부 연속조회 불필요**(응답 tr_cont='' 실측). 단건/랭킹 스냅샷이라 1콜로 끝
- 일반 규약(참고 【문서】): 요청 tr_cont 최초=''·다음='N', 응답 tr_cont F/M=다음 있음·D/E=마지막. 연속 시 body의 `ctx_area_fk100/nk100`(대량 TR은 fk200/nk200)을 다음 요청에 그대로 실어 while 루프
- **페이지네이션이 실제로 필요한 곳은 별개 TR** — 회원사 실시간 매매동향 틱(`frgnmem-trade-trend`)만 output1/output2 + tr_cont 연속조회 사용. 본 스펙 대상 아님

## 3. PIT(그 시점에 알 수 있었나) 주의점

### 3.1 잠정수급 갱신 슬롯 — 15:10 잡의 구조적 공백
- 추정수급(1.2)·외인기관 랭킹(1.3)·회원사(1.4) 모두 **잠정 가집계**. 입력 슬롯 외국인 09:30/11:20/13:20/14:30, 기관 10:00/11:20/13:20/14:30(±10분) 【문서】
- **15:10 잡 시점의 최신 값 = 14:30 슬롯** → **14:30~15:30(연속매매 마지막 1시간 + 종가 동시호가)는 아예 미반영**. 이 공백은 이 API군의 구조적 한계
- ±10분 변동으로 15:10에 14:30 갱신이 아직 안 됐을 수 있으니 **스냅샷 캡처 벽시계 시각을 반드시 함께 저장**(특히 1.4는 행별 타임스탬프가 없어 저장 안 하면 룩어헤드 위험)

### 3.2 호가 시장구분코드(J/NX/UN) 선택
- **1.1 호가: 반드시 `J`(KRX 단독) 하드코딩** 【실측】. 근거: 005930 동일순간 `total_askp_rsqn` J=839,311 + NX=820,551 ≈ UN=1,659,913 → UN은 KRX+NXT 합산이라 **잔량이 약 2배로 부풀어 NXT 오염**. 예상체결도 J·UN=283,500인데 NX=0(NXT는 예상체결 미제공)
- **1.4 회원사: `J` 사용** 【실측】. NX=0건(외국계 가집계에 NXT 피드 없음), UN은 랭킹이 달라 별개 데이터(삼성전자가 top100 이탈)
- 1.2는 시장구분 파라미터 자체가 없고, 1.3은 `V` 고정(KRX 정규시장) — 둘 다 NXT 분리 개념 없음
- **NXT 도입 후 UN이 '기본값'처럼 다뤄질 위험** → 코드에서 J를 명시적으로 고정

### 3.3 예상체결 룩어헤드 함정 (1.1 output2)
- **연속장 중 `antc_cnpr`/`antc_vol`은 09:00 시가단일가 결과의 잔상** 【실측: antc_cnpr 283,500 == stck_oprc 283,500 정확히 일치】
- 즉 **15:10에 antc_cnpr을 읽으면 그것은 '예상 종가'가 아니라 아침 시가다.** 진짜 종가 예상체결가/량은 **15:20~ 마감 동시호가 구간에서 별도 스냅샷**을 떠야 유효
- 설계: **호가창 잡(15:10)과 종가 예상체결 잡(15:20+)을 분리.** 세션 위상은 `new_mkop_cls_code`(연속장=20)·`antc_mkop_cls_code`(연속장=112)로 확인, 동시호가 진입 시 값이 바뀜 【실측】

### 3.4 forward-only 적재 원칙
- 1.2·1.3·1.4 잠정치는 **당일 저녁 확정치로 덮여 보존되지 않음** → 과거 재구성 불가. **지금부터 매일 15:10(및 15:20) 스냅샷 적재만이 백테스트 원천** 【문서】
- **백테스트에는 절대 투입 금지** — 백테스트 수급은 KRX 확정치(pykrx EOD, T-1)만 사용. 잠정↔확정 혼용은 (1)룩어헤드 (2)추정-확정 괴리 이중 오염(메모리 방침·CLAUDE.md 룩어헤드 원칙과 정합)

## 4. 실측·문서 불일치 및 미해결 항목

1. **FID_INPUT_ISCD 코스닥 코드가 엔드포인트마다 다름** ⚠️
   - 1.3 `foreign-institution-total`: 코스닥=**1001** 【실측: iscd=1001에서 에코프로 086520·에코프로비엠 247540 반환】
   - 1.4 `frgnmem-trade-estimate`: 코스피=**1001**, 코스닥=**2001** 【문서·미검증 — 1.4의 KOSDAQ 필터는 실측 미확인】
   - 같은 '1001'이 1.3에선 코스닥, 1.4에선 코스피로 충돌. **엔드포인트별 코드표를 분리**하고 **1.4의 1001/2001은 15:10 착수 전 실측 검증 필요**

2. **예상체결 '수량' 필드명 함정**: 1.1의 수량은 `antc_vol`이며 통념의 `antc_cnqn`은 이 TR 응답에 **아예 없음** 【실측】. 파서 필드명 혼동 주의

3. **`glob_ntsl_qty` 이름 함정**: 1.4 필드명은 순'매도'인데 **양수=순매수** 【실측】. 부호 해석 반대로 하면 신호 뒤집힘

4. **버킷→벽시계 슬롯 매핑 미확정**: 1.2 `bsop_hour_gb`는 오전 단일 관측이라 **"최고 인덱스=최신, 종가 최종=14:30"만 확정** 【실측】. 중간 슬롯(09:30/10:00/11:20/13:20) 정확 매핑은 미확정 → **수집기는 벽시계 슬롯이 아니라 max 인덱스 행을 사용해 회피**. 완전 확정하려면 13:20 이후·15:10 재관측 필요

5. **잠정→확정 정정이 순매수 부호를 뒤집는지 미확인** 【미검증】: 1.2·1.3·1.4 모두 신호로 쓰기 전 실측 필요(특히 종가베팅 타깃인 막판 급변 종목에서 추정 오차 최대 가능)

6. **UN(통합) 코드 성격이 엔드포인트마다 다름**: 1.1 UN=J+NX 잔량 합산(예측 가능) 【실측】 vs 1.4 UN=랭킹 재정렬로 종목 구성 자체가 달라짐 【실측】. UN을 범용으로 신뢰하지 말 것

7. **1.2 종가 기준 최대 행 수 미확정**: 오전 3행 실측, 종가 최대 4~5행은 예상치 【문서·미검증】

8. **1.4 회원사 데이터는 '투자자 유형' 아님**: 외국계 창구 프록시라 실제 외국인 순매수와 개념·포괄범위가 다름(창구 기준). 1.2(투자자 유형 추정)와 값 성격이 다르므로 **동일 지표로 합치거나 상호 검산하지 말 것** — 방향성 참고용 선행지표

9. **토큰 분당 1회 발급 제한** 【문서】: 만료 임박 재발급 실패 리스크 → 장전 선발급으로 회피(§2.1)

10. **`rt_cd=0` + output 0건 케이스**(NX에서의 1.1 예상체결·1.4): 오류가 아니라 정상 응답이므로 에러 재시도 로직이 빈 응답을 유량 낭비 재시도로 처리하지 않도록 분기 필요 【실측】
